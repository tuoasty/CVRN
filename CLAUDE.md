## Project

CVRN (Confederate Volleyball Roblox Network) — a Next.js 15 full-stack web app for managing a Roblox volleyball league. Features public pages (standings, matches, teams, playoffs) and an admin dashboard for league management.

**Tech stack:** Next.js 15, React 19, TypeScript 5, Tailwind CSS, SWR, Zustand, Supabase, Pino, Vercel.

## Commands

```bash
npm run dev      # Start development server
npm run build    # Production build
npm run lint     # ESLint
```

There are no tests configured.

## Architecture

4-layer architecture — each layer only calls the one below it:

```
Server Action (app/actions/*.actions.ts)
  → Service (server/services/*.service.ts)
  → Repository (server/db/*.repo.ts)
  → Supabase (PostgreSQL via PostgREST)
```

**Client-side data fetching** uses SWR hooks in `app/hooks/use*.ts`. Global UI state (selected region/season) lives in `app/stores/publicContextStore.ts` (Zustand + localStorage).

**Error handling** uses `Result<T, E>` from `shared/types/result.ts` with `Ok()`/`Err()` helpers. Services return `Result` objects; don't throw raw errors. Errors are serialized via `server/utils/serializeableError.ts`.

**Auth** is handled by Supabase SSR middleware (`server/supabase/middleware.ts`). Separate Supabase clients for server, browser, and admin contexts in `server/supabase/`.

**DTOs** in `server/dto/` are TypeScript interfaces only (no runtime validation). Database types are auto-generated into `database.types.ts` (UTF-16 encoded) and overridden in `shared/types/db.override.ts` (BigInt fix for roblox_user_id).

## Key Directories

- `app/actions/` — Server Actions (API layer); one file per domain
- `app/hooks/` — SWR data-fetching hooks (useMatches, useTeams, etc.)
- `app/features/` — Feature-specific UI components (dialogs, forms, tables)
- `app/(public)/` — Public-facing pages (route group)
- `app/admin/` — Admin dashboard pages
- `app/components/ui/` — shadcn/ui primitives
- `app/components/public/` — Public layout components (navbar, footer)
- `server/services/` — Business logic
- `server/db/` — Repository layer; direct Supabase queries, no ORM
- `server/dto/` — Input shapes (TypeScript interfaces)
- `server/roblox/` — Roblox API client
- `shared/types/` — Types shared between client and server

## External Integrations

- **Supabase** — database, auth, file storage (team logos)
- **Roblox API** — player/official profile + avatar fetching (`server/roblox/`)
- **Google Sheets** — data import for league management
- **Vercel** — deployment, analytics (`@vercel/analytics`, `@vercel/speed-insights`)

## Conventions

- Use `cn()` from `lib/utils.ts` for conditional Tailwind class merging
- UI components come from `app/components/ui/` (shadcn/ui, "new-york" style)
- Path alias `@/*` resolves to the project root
- Pino logger is configured in `server/utils/logger.ts`
- File naming: `*.actions.ts`, `*.service.ts`, `*.repo.ts`, `*.dto.ts`, `use*.ts`
- All IDs are UUIDs (string). Roblox user IDs are BigInt stored as string.
- Soft deletes on teams via `deleted_at` column
- Standings are a database view (computed from match results), not a table

## Known Issues

- `standing.action.ts` uses singular naming vs `*.actions.ts` plural pattern elsewhere
- `match.service.ts` is 1,147 lines — candidate for decomposition
- DTOs have no runtime validation (no Zod); type safety is compile-time only
- `database.types.ts` is UTF-16 encoded (auto-generated by Supabase CLI)
- README.md is generic Next.js starter boilerplate, not customized for CVRN
